## Project Overview

**Rohit Vipin Mathews** — Personal portfolio site. Next.js 16 static export deployed to GitHub Pages. Data-driven (JSON headless CMS pattern). UI pulls all content from JSON files — never hardcoded.

- **Live:** https://rohitvipin.github.io/
- **Repo:** https://github.com/rohitvipin/rohitvipin.github.io
- **Stack:** Next.js 16.2.4, React 19.2.4, TypeScript 5, Tailwind 4, Vitest 4

## Repository Structure

- **`data/`** — Canonical JSON content (source of truth)
  - Arrays: `awards.json`, `community.json`, `education.json`, `experience.json`, `nav.json`, `projects.json`, `skills.json`, `socials.json`
  - Objects: `leadership.json` (title + sections), `profile.json` (singleton)
  - `resume-config.json` — PDF layout config, consumed only by `utils/generate-resume.ts` (not loaded by `data.ts`, not in Zod schemas)
- **`src/`** — Next.js App Router + React components
  - `app/` — Layout, pages, global styles
  - `components/` — Section and shared components (organized by feature folder)
  - `lib/` — `data.ts` (typed constants, one per data file), `colors.ts` (company/domain color utils), `schemas.ts` (Zod validation schemas)
  - `types/` — TypeScript interfaces (must stay in sync with `data/*.json`)
  - `__tests__/` — Unit + integration tests (Vitest + React Testing Library)
- **`utils/`** — Dev scripts (run via `tsx`)
  - `generate-favicons.ts` — Favicon + OG image generation
  - `generate-resume.ts` — PDF resume via `@react-pdf/renderer`
  - `lint-data.ts`, `lint-data-core.ts` — JSON schema validation (uses Zod schemas from `src/lib/schemas.ts`)
- **`public/`** — Static assets (robots.txt, sitemap.xml, og-image, favicon suite)
- **`docs/`** — Architecture + content strategy specs (CODE_REVIEW.md, DEVELOPMENT.md, CONTRIBUTING.md, etc.)
- **`.github/workflows/`** — Two workflows: `ci.yml` (lint → test → build on PRs), `deploy.yml` (lint → test → generate-resume → build → deploy to gh-pages on push to main)

## Resume PDF — Layout Safety

The PDF resume at `public/Rohit_Vipin_Mathews_Resume.pdf` is generated by `utils/generate-resume.ts` via `npm run generate-resume` (runs in `deploy.yml` before every deploy).

**When changing `src/app/page.tsx`, `src/components/`, or `data/*.json`:**

1. Run `npm run generate-resume` locally and visually inspect the output PDF.
2. Check: section headers, headline text, metrics line, tech stacks, and page breaks.
3. The resume uses `data/leadership.json` title, `profile.headline`, and `profile.key_metrics` (primary tier only) — changes to these fields affect both the site and the PDF.
4. `data/impact.json` is **not** used by the resume generator — impact stories are web-only.

## Architecture Principles

### 1. Headless CMS Pattern

- **Source of truth:** `data/*.json` files only.
- **Never hardcode content** into components, styles, or templates.
- **UI components** are data-driven — import data via typed getters in `src/lib/data.ts`.
- **Schema changes:** New JSON fields require matching updates to `src/types/index.ts` and `src/lib/schemas.ts` in the same commit.

### 2. Type Safety & Validation

- **Strict TypeScript** — no `any` types (use generics or discriminated unions).
- **All JSON schemas** mirror TypeScript interfaces in `src/types/index.ts`.
- **Zod schemas** in `src/lib/schemas.ts` validate JSON at build time via `npm run lint:data`.
- **Build-time data loading** — no `fetch()` in client code. Data imported at build time via Next.js SSG.
- **Discriminated unions** for conditional props (not booleans).

### 3. Component Architecture

- **Server components by default** — add `"use client"` only for hooks/browser APIs.
- **Semantic naming** — PascalCase for components, camelCase for utilities.
- **Props interfaces** — always defined and exported, even for internal components.
- **No prop drilling** — use context for nesting beyond 1 level.

### 4. Styling & Theming

- **CSS custom properties** — all colors use the design tokens below. No hardcoded hex/rgb.
- **Define tokens** in `src/app/globals.css` under `:root` (dark default) and `[data-theme="light"]` (light theme), matching `next-themes` `attribute="data-theme"` config.
- **Tailwind 4** for structure (layout, spacing, utilities). No `tailwind.config.*` — uses `@tailwindcss/postcss`.
- **Icons** — `react-icons` only (FA6 brands or Feather UI). No inline SVGs.
- **Theme switching** — `next-themes` manages dark/light mode.

**Available CSS tokens (defined in `globals.css`):**

| Token           | Purpose                     |
| --------------- | --------------------------- |
| `--bg`          | Page background             |
| `--surface`     | Card / section background   |
| `--surface-2`   | Elevated surface            |
| `--border`      | Borders and dividers        |
| `--accent`      | Primary accent colour       |
| `--accent-glow` | Accent with glow effect     |
| `--accent-2`    | Secondary accent            |
| `--text`        | Primary text                |
| `--muted`       | Secondary text              |
| `--muted-2`     | Tertiary / placeholder text |
| `--success`     | Success state               |
| `--shadow-card` | Card shadow                 |

### 5. Testing

- **Vitest** for all tests (fast, ESM-native, jsdom environment).
- **Minimum coverage:** 60% all metrics (enforced via `vitest.config.ts` thresholds).
- **100% coverage** for utilities in `src/lib/`.
- **New components:** test main render paths and key interactions.
- **Semantic assertions** — test **what** the user sees, not **how** the component works internally.
- **Mock data** — never read real `data/*.json` in tests; supply fixtures inline.

### 6. Static Export

- **No Node.js runtime** — static export to GitHub Pages (`output: "export"` in `next.config.mjs`).
- **No Node.js imports** in client code (no `fs`, `path`, etc.).
- **All dynamic data** must exist at build time (embedded in JSON files).
- `images: { unoptimized: true }` and `trailingSlash: true` set in `next.config.mjs`.

## Path-Specific Rules

### `data/` — JSON Content Files

**Rules:**

- ✓ Arrays of objects, or top-level object for singletons (`profile.json`, `leadership.json`, `resume-config.json`).
- ✓ All strings use **plain ASCII punctuation** only:
  - No smart quotes (`"` → `"`), no em/en dashes (`–`/`—` → `-`), no ellipsis (`…` → `...`)
- ✓ **British English:** -ise not -ize, -our not -or, organised not organized. Exception: proper nouns, product names.
- ✓ **No corporate jargon:** "world-class", "exceptional", "cutting-edge", "seamlessly", "end-to-end", "innovative", "robust", "proactive" are AI tells — rewrite.
- ✓ Required fields must be present in all records.
- ✓ IDs and slugs must be unique across the entire dataset.
- ✓ Validate before commit: `npm run lint:data`

### `src/types/index.ts` — TypeScript Interfaces

**Exported types:** `KeyMetric`, `Profile`, `ExperienceEntry`, `Product`, `Project`, `SkillCategory`, `Education`, `Social`, `Award`, `CommunityEntry`, `LeadershipSubsection`, `Leadership`, `NavLink`, `ResumeSectionConfig`, `ResumeConfig`

**Rules:**

- ✓ Interface name matches JSON file concept (e.g., `SkillCategory` for `skills.json` items).
- ✓ **Always update when `data/*.json` changes** (same commit as schema + data change).
- ✓ All JSON fields represented (required or `?` optional).
- ✓ No `any`, no `Record<string, unknown>` without justification.
- ✓ Export all types used by components.

### `src/lib/` — Data Loaders, Utilities & Schemas

**Files:**

- `data.ts` — imports all JSON, exports typed constants: `profile`, `experience`, `projects`, `skills`, `education`, `socials`, `awards`, `community`, `leadership`, `navLinks` (each parsed via Zod at import time)
- `colors.ts` — `getCompanyColor()`, `getDomainColor()` (pure functions)
- `schemas.ts` — Zod schemas for all data types; shared by `lint-data.ts` and test suite

**Rules:**

- ✓ `data.ts` is **build-time only** — no dynamic imports, no `fetch()`.
- ✓ Functions fully typed (no `any`, explicit return types).
- ✓ Adding a new data file → add Zod schema to `schemas.ts`, getter to `data.ts`, type to `types/index.ts`.
- ✓ 100% test coverage for all `src/lib/` files.

### `src/components/` — React Components

**Feature folders:** `about/`, `awards/`, `community/`, `education/`, `experience/`, `hero/`, `leadership/`, `projects/`, `skills/`, `shared/`

**Shared components:** `AnimateOnScroll`, `Nav`, `ScrollToTop`, `SectionHeader`, `SocialLinks`, `TechChip`, `ThemeToggle`

**Rules:**

- ✓ **Server components by default** — no `"use client"` unless using hooks or browser APIs.
- ✓ `"use client"` required for: `ThemeToggle`, `ScrollToTop`, `Nav`, `AnimateOnScroll` (interactive/browser-dependent).
- ✓ Props interface always defined and exported.
- ✓ **No hardcoded strings** — use typed data loaders.
- ✓ **CSS tokens only** — use `var(--accent)`, `var(--surface)`, etc. Never hardcode colours.
- ✓ **Icons from `react-icons`** — no inline SVGs.
- ✓ Semantic HTML (`button`, `a`, `nav`, `section`, `article`, etc.).

### `src/__tests__/` — Tests

**Rules:**

- ✓ File name matches source file (e.g., `Nav.test.tsx` for `Nav.tsx`).
- ✓ Vitest + React Testing Library.
- ✓ Semantic test names — describe **what** is rendered/behaviour, not implementation.
- ✓ Tests must pass locally before commit (`npm run test`).
- ✓ Mock external data; never rely on real `data/*.json` files in component tests.

**Example:**

```typescript
import { describe, it, expect, vi } from "vitest";
import { render, screen } from "@testing-library/react";
import { SectionHeader } from "@/components/shared/SectionHeader";

describe("SectionHeader", () => {
  it("renders title and subtitle", () => {
    render(<SectionHeader title="Skills" subtitle="What I work with" />);
    expect(screen.getByText("Skills")).toBeInTheDocument();
  });
});
```

### `utils/` — Dev Scripts

**Rules:**

- ✓ Run via `tsx utils/script-name.ts`.
- ✓ One-off scripts, not library code.
- ✓ Exit with status 1 on error.
- ✓ Output to stdout/stderr, not files (except `generate-resume.ts` and `generate-favicons.ts` which write to `public/`).

### `.github/workflows/` — CI/CD

**Two workflows:**

- **`ci.yml`** — triggers on PRs: Prettier → ESLint → Vitest → Next.js build
- **`deploy.yml`** — triggers on push to `main`: Prettier → ESLint → Vitest → generate-resume → Next.js build → deploy to `gh-pages`

**Rules:**

- ✓ Node version locked to 20.x.
- ✓ Pipeline fails if any step fails.
- ✓ `NEXT_PUBLIC_BASE_PATH` set in workflow (currently `""`).
- ✓ `.nojekyll` added to output before deploy.

## Quick Reference: Adding a Feature

1. **Add JSON** to `data/feature.json` → validate with `npm run lint:data`.
2. **Add Zod schema** to `src/lib/schemas.ts`.
3. **Define type** in `src/types/index.ts`.
4. **Add getter** to `src/lib/data.ts`.
5. **Build component** in `src/components/feature/`.
6. **Add tests** in `src/__tests__/`.
7. **Check build** with `npm run build`.
8. **Push PR** — `ci.yml` validates lint, tests, and build.
