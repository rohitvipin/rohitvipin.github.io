## Project Overview

**Rohit Vipin Mathews** тАФ Personal portfolio site. Next.js 16 static export deployed to GitHub Pages. Data-driven (JSON headless CMS pattern). UI pulls all content from JSON files тАФ never hardcoded.

- **Live:** https://rohitvipin.github.io/
- **Repo:** https://github.com/rohitvipin/rohitvipin.github.io
- **Stack:** Next.js 16.2.4, React 19.2.5, TypeScript 6, Tailwind 4, Vitest 4

## Repository Structure

- **`data/`** тАФ Canonical JSON content (source of truth)
  - Arrays: `awards.json`, `community.json`, `education.json`, `experience.json`, `nav.json`, `projects.json`, `skills.json`, `socials.json`
  - Objects: `leadership.json` (title + sections), `profile.json` (singleton)
  - `resume-config.json` тАФ PDF layout config, consumed only by `utils/generate-resume.ts` (not loaded by `data.ts`; validated by `ResumeConfigSchema` in `src/lib/schemas.ts` via `npm run lint:data`)
- **`src/`** тАФ Next.js App Router + React components
  - `app/` тАФ Layout, pages, global styles
  - `components/` тАФ Section and shared components (organized by feature folder)
  - `lib/` тАФ `data.ts` (typed constants, one per data file), `colors.ts` (company/domain color utils), `schemas.ts` (Zod validation schemas), `jsonld.ts` (JSON-LD structured data builder)
  - `types/` тАФ TypeScript interfaces (must stay in sync with `data/*.json`)
  - `__tests__/` тАФ Unit + integration tests (Vitest + React Testing Library)
- **`utils/`** тАФ Dev scripts (run via `tsx`)
  - `generate-favicons.ts` тАФ Favicon + OG image generation
  - `generate-resume.ts` тАФ PDF resume via `@react-pdf/renderer`
  - `generate-sitemap.ts` тАФ Sitemap XML; `lastmod` derived from `git log` on `data/` so it reflects actual content changes, not build date. Exports `buildSitemap(baseUrl, lastmod)` for unit testing.
  - `fetch-avatar.ts` тАФ Fetches GitHub avatar with SHA-256 pinning, magic-byte validation, size limits, and `https:`-only enforcement
  - `lint-data.ts`, `lint-data-core.ts` тАФ JSON schema validation (uses Zod schemas from `src/lib/schemas.ts`)
- **`public/`** тАФ Static assets (robots.txt, sitemap.xml, og-image, favicon suite)
- **`docs/`** тАФ Architecture + content strategy specs (CODE_REVIEW.md, DEVELOPMENT.md, CONTRIBUTING.md, etc.)
- **`.github/workflows/`** тАФ Three workflows: `ci.yml` (PR gate: dep-review + gitleaks scans, then calls reusable), `deploy.yml` (push to main: calls reusable with lighthouse + pages upload, then deploy + smoke-test jobs), `_lint-test-build.yml` (reusable: lint тЖТ test тЖТ prepare-assets тЖТ build тЖТ cached e2e, optional lighthouse/pages-upload via inputs)

## Resume PDF тАФ Layout Safety

The PDF resume at `public/Rohit_Vipin_Mathews_Resume.pdf` is generated by `utils/generate-resume.ts` via `npm run generate-resume`. It runs as part of the `prepare-assets` composite action invoked by the reusable `_lint-test-build.yml` workflow, so every PR build and every main deploy regenerate it.

**When changing `src/app/page.tsx`, `src/components/`, or `data/*.json`:**

1. Run `npm run generate-resume` locally and visually inspect the output PDF.
2. Check: section headers, headline text, metrics line, tech stacks, and page breaks.
3. The resume uses `data/leadership.json` title, `profile.headline`, and `profile.key_metrics` (primary tier only) тАФ changes to these fields affect both the site and the PDF.
4. `data/impact.json` is **not** used by the resume generator тАФ impact stories are web-only.

## Architecture Principles

### 1. Headless CMS Pattern

- **Source of truth:** `data/*.json` files only.
- **Never hardcode content** into components, styles, or templates.
- **UI components** are data-driven тАФ import data via typed getters in `src/lib/data.ts`.
- **Schema changes:** New JSON fields require matching updates to `src/types/index.ts` and `src/lib/schemas.ts` in the same commit.

### 2. Type Safety & Validation

- **Strict TypeScript** тАФ no `any` types (use generics or discriminated unions).
- **All JSON schemas** mirror TypeScript interfaces in `src/types/index.ts`.
- **Zod schemas** in `src/lib/schemas.ts` validate JSON at build time via `npm run lint:data`.
- **Build-time data loading** тАФ no `fetch()` in client code. Data imported at build time via Next.js SSG.
- **Discriminated unions** for conditional props (not booleans).

### 3. Component Architecture

- **Server components by default** тАФ add `"use client"` only for hooks/browser APIs.
- **Semantic naming** тАФ PascalCase for components, camelCase for utilities.
- **Props interfaces** тАФ always defined and exported, even for internal components.
- **No prop drilling** тАФ use context for nesting beyond 1 level.

### 4. Styling & Theming

- **CSS custom properties** тАФ all colors use the design tokens below. No hardcoded hex/rgb.
- **Define tokens** in `src/app/globals.css` under `:root` (dark default) and `[data-theme="light"]` (light theme), matching `next-themes` `attribute="data-theme"` config.
- **Tailwind 4** for structure (layout, spacing, utilities). No `tailwind.config.*` тАФ uses `@tailwindcss/postcss`.
- **Icons** тАФ `react-icons` only (FA6 brands or Feather UI). No inline SVGs.
- **Theme switching** тАФ `next-themes` manages dark/light mode.

**Available CSS tokens (defined in `globals.css`):**

| Token           | Purpose                     |
| --------------- | --------------------------- |
| `--bg`          | Page background             |
| `--surface`     | Card / section background   |
| `--surface-2`   | Elevated surface            |
| `--border`      | Borders and dividers        |
| `--accent`      | Primary accent colour       |
| `--accent-glow` | Accent with glow effect     |
| `--accent-2`    | Secondary accent            |
| `--text`        | Primary text                |
| `--muted`       | Secondary text              |
| `--muted-2`     | Tertiary / placeholder text |
| `--success`     | Success state               |
| `--shadow-card` | Card shadow                 |

### 5. Testing

- **Vitest** for all tests (fast, ESM-native, jsdom environment).
- **Minimum coverage:** 90% statements/functions/lines, 85% branches (enforced via `vitest.config.ts` thresholds).
- **100% coverage** for utilities in `src/lib/`.
- **New components:** test main render paths and key interactions.
- **Semantic assertions** тАФ test **what** the user sees, not **how** the component works internally.
- **Mock data** тАФ never read real `data/*.json` in tests; supply fixtures inline.

### 6. Static Export

- **No Node.js runtime** тАФ static export to GitHub Pages (`output: "export"` in `next.config.mjs`).
- **No Node.js imports** in client code (no `fs`, `path`, etc.).
- **All dynamic data** must exist at build time (embedded in JSON files).
- `images: { unoptimized: true }` and `trailingSlash: true` set in `next.config.mjs`.

### 7. UI Validation

**Every UI change тАФ bug fix, feature, style tweak, content update тАФ must be verified in a real browser before marking done.**

**Mandatory process:**

1. Start dev server: `npm run dev` (http://localhost:3000).
2. Use **Playwright MCP** (`mcp__playwright__*`) or **Chrome DevTools MCP** (`mcp__chrome-devtools__*`) to navigate, screenshot, and interact.
3. Verify the changed section renders correctly in both **dark and light themes** (toggle via the theme button).
4. Check **mobile viewport** (375px width minimum) and **desktop** (1280px+).
5. Confirm no console errors or layout breakage in adjacent sections.
6. For interactive elements (nav, scroll-to-top, collapsible cards): exercise the interaction and screenshot the result.

**Tool preference:**

- `mcp__playwright__browser_navigate` тЖТ `mcp__playwright__browser_take_screenshot` for visual spot-checks.
- `mcp__playwright__browser_snapshot` for accessibility tree / element assertions.
- `mcp__chrome-devtools__lighthouse_audit` for performance/a11y regressions on significant changes.
- `mcp__playwright__browser_console_messages` to catch runtime JS errors post-change.

**Never claim UI work is complete without at least one screenshot from a real browser session.**

## Path-Specific Rules

### `data/` тАФ JSON Content Files

**Rules:**

- тЬУ Arrays of objects, or top-level object for singletons (`profile.json`, `leadership.json`, `resume-config.json`).
- тЬУ All strings use **plain ASCII punctuation** only:
  - No smart quotes (`"` тЖТ `"`), no em/en dashes (`тАУ`/`тАФ` тЖТ `-`), no ellipsis (`тАж` тЖТ `...`)
- тЬУ **British English:** -ise not -ize, -our not -or, organised not organized. Exception: proper nouns, product names.
- тЬУ **No corporate jargon:** "world-class", "exceptional", "cutting-edge", "seamlessly", "end-to-end", "innovative", "robust", "proactive" are AI tells тАФ rewrite.
- тЬУ Required fields must be present in all records.
- тЬУ IDs and slugs must be unique across the entire dataset.
- тЬУ Validate before commit: `npm run lint:data`
- тЬУ **`duration` field** тАФ three allowed formats (used in `experience.json` and `projects.json`):
  - `"Month YYYY - Present"` тАФ e.g. `"April 2024 - Present"`
  - `"Month YYYY - Month YYYY"` тАФ e.g. `"January 2012 - July 2013"`
  - `"YYYY"` тАФ bare year for short-run items e.g. `"2021"`
  - Parsed by `src/lib/duration.ts:parseStartYear` to derive sort order; do not use other formats.

### `src/types/index.ts` тАФ TypeScript Interfaces

**Exported types:** `KeyMetric`, `ValueProposition`, `Profile`, `ImpactStory`, `ExperienceEntry`, `Product`, `Project`, `SkillCategory`, `Education`, `Social`, `Award`, `CommunityEntry`, `LeadershipSubsection`, `Leadership`, `NavLink`, `ResumeSectionConfig`, `ResumeConfig`

**Rules:**

- тЬУ Interface name matches JSON file concept (e.g., `SkillCategory` for `skills.json` items).
- тЬУ **Always update when `data/*.json` changes** (same commit as schema + data change).
- тЬУ All JSON fields represented (required or `?` optional).
- тЬУ No `any`, no `Record<string, unknown>` without justification.
- тЬУ Export all types used by components.

### `src/lib/` тАФ Data Loaders, Utilities & Schemas

**Files:**

- `data.ts` тАФ imports all JSON, exports typed constants: `profile`, `experience`, `projects`, `skills`, `education`, `socials`, `awards`, `community`, `leadership`, `navLinks`, `impact` (each parsed via Zod at import time)
- `colors.ts` тАФ `getCompanyColor()`, `getDomainColor()` (pure functions)
- `schemas.ts` тАФ Zod schemas for all data types; shared by `lint-data.ts` and test suite
- `jsonld.ts` тАФ `buildPersonJsonLd()` builds JSON-LD `Person` structured data from profile + socials; used by `src/app/layout.tsx`

**Rules:**

- тЬУ `data.ts` is **build-time only** тАФ no dynamic imports, no `fetch()`.
- тЬУ Functions fully typed (no `any`, explicit return types).
- тЬУ Adding a new data file тЖТ add Zod schema to `schemas.ts`, getter to `data.ts`, type to `types/index.ts`.
- тЬУ 100% test coverage for all `src/lib/` files.

### `src/components/` тАФ React Components

**Feature folders:** `about/`, `awards/`, `community/`, `education/`, `experience/`, `hero/`, `leadership/`, `projects/`, `skills/`, `shared/`

**Shared components:** `Nav`, `ScrollToTop`, `SectionHeader`, `SocialLinks`, `TechChip`, `ThemeToggle`

**Rules:**

- тЬУ **Server components by default** тАФ no `"use client"` unless using hooks or browser APIs.
- тЬУ `"use client"` required for: `ThemeToggle`, `ScrollToTop`, `Nav` (interactive/browser-dependent).
- тЬУ **Named exports only** тАФ no `export default`. Use `export function` or `export const`.
- тЬУ **Collapsible content** тАФ use native `<details>/<summary>` with class `card-details`. No custom open/close state.
- тЬУ Props interface always defined and exported.
- тЬУ **No hardcoded strings** тАФ use typed data loaders.
- тЬУ **CSS tokens only** тАФ use `var(--accent)`, `var(--surface)`, etc. Never hardcode colours.
- тЬУ **Icons from `react-icons`** тАФ no inline SVGs.
- тЬУ Semantic HTML (`button`, `a`, `nav`, `section`, `article`, etc.).

### `src/__tests__/` тАФ Tests

**Rules:**

- тЬУ File name matches source file (e.g., `Nav.test.tsx` for `Nav.tsx`).
- тЬУ Vitest + React Testing Library.
- тЬУ Semantic test names тАФ describe **what** is rendered/behaviour, not implementation.
- тЬУ Tests must pass locally before commit (`npm run test`).
- тЬУ Mock external data; never rely on real `data/*.json` files in component tests.

**Example:**

```typescript
import { describe, it, expect, vi } from "vitest";
import { render, screen } from "@testing-library/react";
import { SectionHeader } from "@/components/shared/SectionHeader";

describe("SectionHeader", () => {
  it("renders title and subtitle", () => {
    render(<SectionHeader title="Skills" subtitle="What I work with" />);
    expect(screen.getByText("Skills")).toBeInTheDocument();
  });
});
```

### `utils/` тАФ Dev Scripts

**Rules:**

- тЬУ Run via `tsx utils/script-name.ts`.
- тЬУ One-off scripts, not library code.
- тЬУ Exit with status 1 on error.
- тЬУ Output to stdout/stderr, not files (except `generate-resume.ts` and `generate-favicons.ts` which write to `public/`).

### `.github/workflows/` тАФ CI/CD

**Three workflows:**

- **`ci.yml`** тАФ triggers on PRs: `quality-scans` job (dependency-review + gitleaks) runs in parallel with `lint-test-build` job that calls the reusable workflow.
- **`deploy.yml`** тАФ triggers on push to `main`: `build` job calls the reusable workflow with `lighthouse: true` and `upload-pages: true`; followed by `deploy` (GitHub Pages) and `smoke-test` jobs.
- **`_lint-test-build.yml`** тАФ reusable (`workflow_call`): Prettier тЖТ ESLint тЖТ Vitest тЖТ prepare-assets composite тЖТ Next.js build тЖТ cached Playwright e2e тЖТ optional Lighthouse тЖТ optional pages-artifact upload. Inputs: `audit-level`, `build-sha`, `report-name`, `lighthouse`, `upload-pages`.

**Rules:**

- тЬУ Node version locked to 24.
- тЬУ Pipeline fails if any step fails.
- тЬУ `NEXT_PUBLIC_BASE_PATH` set in workflow (currently `""`).
- тЬУ `.nojekyll` added to output before deploy (gated by `upload-pages` input).
- тЬУ Pipeline body lives in the reusable workflow only тАФ never duplicate steps in `ci.yml` or `deploy.yml`. PR-only concerns (dep-review, gitleaks) stay inline in `ci.yml`.

## Quick Reference: Adding a Feature

1. **Add JSON** to `data/feature.json` тЖТ validate with `npm run lint:data`.
2. **Add Zod schema** to `src/lib/schemas.ts`.
3. **Define type** in `src/types/index.ts`.
4. **Add getter** to `src/lib/data.ts`.
5. **Build component** in `src/components/feature/`.
6. **Add tests** in `src/__tests__/`.
7. **Check build** with `npm run build`.
8. **UI validate** тАФ run `npm run dev`, use Playwright/Chrome DevTools MCP to screenshot dark + light themes, mobile + desktop viewports, check console errors.
9. **Push PR** тАФ `ci.yml` validates lint, tests, and build.
