The PDF resume at `public/Rohit_Vipin_Mathews_Resume.pdf` is generated by `utils/generate-resume.ts` via `npm run generate-resume` (runs in `deploy.yml` before every deploy).

**When changing `src/app/page.tsx`, `src/components/`, or `data/*.json`:**

1. Run `npm run generate-resume` locally and visually inspect the output PDF.
2. Check: section headers, headline text, metrics line, tech stacks, and page breaks.
3. The resume uses `data/leadership.json` title, `profile.headline`, and `profile.key_metrics` (primary tier only) â€” changes to these fields affect both the site and the PDF.
4. `data/impact.json` is **not** used by the resume generator â€” impact stories are web-only.

## Architecture Principles

### 1. Headless CMS Pattern

- **Source of truth:** `data/*.json` files only.
- **Never hardcode content** into components, styles, or templates.
- **UI components** are data-driven â€” import data via typed getters in `src/lib/data.ts`.
- **Schema changes:** New JSON fields require matching updates to `src/types/index.ts` and `src/lib/schemas.ts` in the same commit.

### 2. Type Safety & Validation

- **Strict TypeScript** â€” no `any` types (use generics or discriminated unions).
- **All JSON schemas** mirror TypeScript interfaces in `src/types/index.ts`.
- **Zod schemas** in `src/lib/schemas.ts` validate JSON at build time via `npm run lint:data`.
- **Build-time data loading** â€” no `fetch()` in client code. Data imported at build time via Next.js SSG.
- **Discriminated unions** for conditional props (not booleans).

### 3. Component Architecture

- **Server components by default** â€” add `"use client"` only for hooks/browser APIs.
- **Semantic naming** â€” PascalCase for components, camelCase for utilities.
- **Props interfaces** â€” always defined and exported, even for internal components.
- **No prop drilling** â€” use context for nesting beyond 1 level.

### 4. Styling & Theming

- **CSS custom properties** â€” all colors use the design tokens below. No hardcoded hex/rgb.
- **Define tokens** in `src/app/globals.css` under `:root` (dark default) and `[data-theme="light"]` (light theme), matching `next-themes` `attribute="data-theme"` config.
- **Tailwind 4** for structure (layout, spacing, utilities). No `tailwind.config.*` â€” uses `@tailwindcss/postcss`.
- **Icons** â€” `react-icons` only (FA6 brands or Feather UI). No inline SVGs.
- **Theme switching** â€” `next-themes` manages dark/light mode.

**Available CSS tokens (defined in `globals.css`):**

| Token           | Purpose                     |
| --------------- | --------------------------- |
| `--bg`          | Page background             |
| `--surface`     | Card / section background   |
| `--surface-2`   | Elevated surface            |
| `--border`      | Borders and dividers        |
| `--accent`      | Primary accent colour       |
| `--accent-glow` | Accent with glow effect     |
| `--accent-2`    | Secondary accent            |
| `--text`        | Primary text                |
| `--muted`       | Secondary text              |
| `--muted-2`     | Tertiary / placeholder text |
| `--success`     | Success state               |
| `--shadow-card` | Card shadow                 |

### 5. Testing

- **Vitest** for all tests (fast, ESM-native, jsdom environment).
- **Minimum coverage:** 90% statements/functions/lines, 85% branches (enforced via `vitest.config.ts` thresholds).
- **100% coverage** for utilities in `src/lib/`.
- **New components:** test main render paths and key interactions.
- **Semantic assertions** â€” test **what** the user sees, not **how** the component works internally.
- **Mock data** â€” never read real `data/*.json` in tests; supply fixtures inline.

### 6. Static Export

- **No Node.js runtime** â€” static export to GitHub Pages (`output: "export"` in `next.config.mjs`).
- **No Node.js imports** in client code (no `fs`, `path`, etc.).
- **All dynamic data** must exist at build time (embedded in JSON files).
- `images: { unoptimized: true }` and `trailingSlash: true` set in `next.config.mjs`.

## Path-Specific Rules

### `data/` â€” JSON Content Files

**Rules:**

- âś“ Arrays of objects, or top-level object for singletons (`profile.json`, `leadership.json`, `resume-config.json`).
- âś“ All strings use **plain ASCII punctuation** only:
  - No smart quotes (`"` â†’ `"`), no em/en dashes (`â€“`/`â€”` â†’ `-`), no ellipsis (`â€¦` â†’ `...`)
- âś“ **British English:** -ise not -ize, -our not -or, organised not organized. Exception: proper nouns, product names.
- âś“ **No corporate jargon:** "world-class", "exceptional", "cutting-edge", "seamlessly", "end-to-end", "innovative", "robust", "proactive" are AI tells â€” rewrite.
- âś“ Required fields must be present in all records.
- âś“ IDs and slugs must be unique across the entire dataset.
- âś“ Validate before commit: `npm run lint:data`
- âś“ **`duration` field** â€” three allowed formats (used in `experience.json` and `projects.json`):
  - `"Month YYYY - Present"` â€” e.g. `"April 2024 - Present"`
  - `"Month YYYY - Month YYYY"` â€” e.g. `"January 2012 - July 2013"`
  - `"YYYY"` â€” bare year for short-run items e.g. `"2021"`
  - Parsed by `src/lib/duration.ts:parseStartYear` to derive sort order; do not use other formats.

### `src/types/index.ts` â€” TypeScript Interfaces

**Exported types:** `KeyMetric`, `ValueProposition`, `Profile`, `ImpactStory`, `ExperienceEntry`, `Product`, `Project`, `SkillCategory`, `Education`, `Social`, `Award`, `CommunityEntry`, `LeadershipSubsection`, `Leadership`, `NavLink`, `ResumeSectionConfig`, `ResumeConfig`

**Rules:**

- âś“ Interface name matches JSON file concept (e.g., `SkillCategory` for `skills.json` items).
- âś“ **Always update when `data/*.json` changes** (same commit as schema + data change).
- âś“ All JSON fields represented (required or `?` optional).
- âś“ No `any`, no `Record<string, unknown>` without justification.
- âś“ Export all types used by components.

### `src/lib/` â€” Data Loaders, Utilities & Schemas

**Files:**

- `data.ts` â€” imports all JSON, exports typed constants: `profile`, `experience`, `projects`, `skills`, `education`, `socials`, `awards`, `community`, `leadership`, `navLinks`, `impact` (each parsed via Zod at import time)
- `colors.ts` â€” `getCompanyColor()`, `getDomainColor()` (pure functions)
- `schemas.ts` â€” Zod schemas for all data types; shared by `lint-data.ts` and test suite
- `jsonld.ts` â€” `buildPersonJsonLd()` builds JSON-LD `Person` structured data from profile + socials; used by `src/app/layout.tsx`

**Rules:**

- âś“ `data.ts` is **build-time only** â€” no dynamic imports, no `fetch()`.
- âś“ Functions fully typed (no `any`, explicit return types).
- âś“ Adding a new data file â†’ add Zod schema to `schemas.ts`, getter to `data.ts`, type to `types/index.ts`.
- âś“ 100% test coverage for all `src/lib/` files.

### `src/components/` â€” React Components

**Feature folders:** `about/`, `awards/`, `community/`, `education/`, `experience/`, `hero/`, `leadership/`, `projects/`, `skills/`, `shared/`

**Shared components:** `Nav`, `ScrollToTop`, `SectionHeader`, `SocialLinks`, `TechChip`, `ThemeToggle`

**Rules:**

- âś“ **Server components by default** â€” no `"use client"` unless using hooks or browser APIs.
- âś“ `"use client"` required for: `ThemeToggle`, `ScrollToTop`, `Nav` (interactive/browser-dependent).
- âś“ **Named exports only** â€” no `export default`. Use `export function` or `export const`.
- âś“ **Collapsible content** â€” use native `<details>/<summary>` with class `card-details`. No custom open/close state.
- âś“ Props interface always defined and exported.
- âś“ **No hardcoded strings** â€” use typed data loaders.
- âś“ **CSS tokens only** â€” use `var(--accent)`, `var(--surface)`, etc. Never hardcode colours.
- âś“ **Icons from `react-icons`** â€” no inline SVGs.
- âś“ Semantic HTML (`button`, `a`, `nav`, `section`, `article`, etc.).

### `src/__tests__/` â€” Tests

**Rules:**

- âś“ File name matches source file (e.g., `Nav.test.tsx` for `Nav.tsx`).
- âś“ Vitest + React Testing Library.
- âś“ Semantic test names â€” describe **what** is rendered/behaviour, not implementation.
- âś“ Tests must pass locally before commit (`npm run test`).
- âś“ Mock external data; never rely on real `data/*.json` files in component tests.

**Rules:**

- âś“ Run via `tsx utils/script-name.ts`.
- âś“ One-off scripts, not library code.
- âś“ Exit with status 1 on error.
- âś“ Output to stdout/stderr, not files (except `generate-resume.ts` and `generate-favicons.ts` which write to `public/`).
